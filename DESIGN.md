# Anatomia — コード解析 × 動作構造 × コード↔仕様 × 決定的キャッシュ

> **Anatomia** (名称確定。略称は `project-codes` で確定予定、`An` を想定)。
> コードを「ドメイン」へ解剖し、仕様と実行時挙動に結びつけ、決定的キャッシュの上で
> **AI のクリーンなコード生成を支える**開発者ツール。
>
> 本書は設計のみ (実装未着手)。特定サービスのリポには属さない (システム自身の設計ホーム
> `Ars/Anatomia/`)。既存資産 (Ergo 等) は統合先候補として §11 で参照する。
>
> **確定事項**: 重心 = AI クリーンコード生成支援 (§1.1 supply→verify ループ) / 形態 = コア lib +
> アダプタ群 (§1.2、v1 = MCP/HTTP) / 用語 = **DAG** はキャッシュ土台 (acyclic 必須)・**KG** は
> その上の派生ビュー (循環可・関係クエリ用)。

## 1. 目的とスコープ

ゲームのコードベースを、次の用途に同時に使える知識構造へ変換する。

- **(a) AI への文脈供給**: 高精度・キャッシュ安定な「ドメイン単位の文脈束」を決定的に組んで渡す。
- **(b) 人間の把握**: 動作構造 (メインループ / システム実行順) と複雑度を可視化する。
- **(c) コード ↔ 仕様 の結合**: どのコードがどの仕様クレームを実装しているかを双方向に引く。

本質は **静的構造 (何のドメインが存在するか) と 動的実行 (今フレーム何が火を噴くか) と 仕様 を、
同一のアンカー ID で縫合すること**。

**主対象**: C++/C# のゲーム基盤 (Pictor/Ergo 系・Unity 系)。
**副対象**: 肥大化した非ゲーム (Memoria 等) にも汎用オントロジーで適用可。

非対象: プロンプト内容のチューニング、モデル選定、本ツール外のビルド/CI 統合。

### 1.1 重心: AI のクリーンなコード生成 (supply → verify ループ)

Anatomia は「AI 用 RAG」ではなく、**AI のコード生成を綺麗に保つ建築規約オラクル**。
文脈を渡すだけでなく、生成の前後で二度効く。

```
① 生成前(steering)  「この機能=ドメインM(契約C)、実装はanchor X,Y、仕様§S、
                     ここの規約はZ、追加すべき綺麗な場所はW」をAIに渡す
② AIがコードを書く   既存ドメインに沿った、逸脱しないコード
③ 生成後(verify)    契約を満たすか / 正しい仕様に結びつくか / 結合度が上がってないか /
                     既存ドメインを重複実装してないか ← 不合格なら具体的違反を返す
④ 再生成            違反付きで②へ。クリーンになるまで
```

→ §4.3 のドメイン契約と §8 の複雑度メトリクスは、検出でなく **能動的なゲート**。
決定的キャッシュのおかげでループで何度叩いても安い。詳細は §9.1。

> **「クリーン」の定義**: 抽象的な美しさではなく、**この codebase の "grain" (ドメイン・規約・仕様) に
> 沿っているか**。低結合・重複なし・既存ドメインとの一貫性。基盤の流儀に従うこと (≠ 綺麗に書くこと)。

### 1.2 形態: コア lib + アダプタ群

特定リポ (Ergo 等) には属さない。AI/CI/人間の全消費者を素直に従える形。

- **コア lib**: Parse / Merkle-AST(DAG) / ドメイン契約 / ドメインカード / KG / 決定的束。全アダプタ共有。
- **v1 アダプタ = MCP/HTTP 口**: AI が生成中に叩く (①供給・③verify)。重心に直結。
- 続く: **CLI ゲート** (CI で③をブロッカーに) / **Web viz** (人間用、Ergo 配信路に相乗り
  = Ergo は出力アダプタの一候補で、本体は持たない)。

## 2. 設計思想 (3 原則)

1. **キャッシュをデータ構造にする。** 後付けの索引でなく、コンテンツアドレッシング (Merkle) で
   無効化を構造から *無料で・正確に* 得る。ビルドキャッシュ (Bazel/Nix) や git と同じ原理。
2. **静的・動的・仕様を同一アンカー ID で縫う。** 「存在するドメイン」「実行されたドメイン」「仕様クレーム」が
   同じ ID で繋がる。
3. **結合は類似度でなく適合・証拠で。** embedding は補助信号に留め、契約 (conformance) と
   明示参照を一級市民にする。

> 重要な一致: 本ツールのキャッシュ無効化 (AST ノード変更 → 祖先を無効化) は、
> LLM プロンプトキャッシュの無効化 (トークン変更 → 後続を無効化) と **同じ法則**。
> 前方一致が「部分木一致」になっただけ (→ `Lapilli/packages/llm-gateway/DESIGN.md`)。

## 3. 全体アーキテクチャ

```
┌── 静的層: コンテンツアドレス DAG ─────────────────────┐
│ Parse(tree-sitter) → Merkle-AST → ドメイン契約 → ドメインカード │
│                          ↕ 3段リンカ                    │
│                       仕様クレーム                      │
│                    └─ KG 射影 (traceability クエリ)     │
└──────────────────────────┬──────────────────────────────┘
                           │ 同一 Anchor ID で縫合
┌──────────────────────────┴── 動的層: 実行トレース ──────┐
│ 静的ループ骨格抽出 → マーカー自動注入 → フレーム×ドメイントレース │
└──────────────────────────┬──────────────────────────────┘
                           │
┌──────────────────────────┴── 配信・可視化層 ────────────┐
│ Web ビュー (タイムライン/光るグラフ/You are here) ＋ AI 文脈束 │
└─────────────────────────────────────────────────────────┘
```

すべてのノード・トレース・仕様は **Anchor ID = 正規化 Merkle ハッシュ** で識別され、層を跨いで結合する。

## 4. 静的層: コンテンツアドレス DAG

### 4.1 Merkle-AST

- `tree-sitter` で AST を抽出し、**各ノードに Merkle ハッシュ**を振る (ノード内容 + 子ハッシュ)。
- 仕様クレームやマーカーは、この **部分木ハッシュ (= Anchor ID) にアンカー**する。
- 関数を 1 個編集 → その部分木と祖先のハッシュだけ変化 → **依存する成果物だけ無効化**。
  スナップショットドメイン・差分索引ドメインは不要。Merkle 構造がそのままバージョニング兼無効化。

### 4.2 正規化ハッシュ (命中率の心臓)

最難所。素朴に AST をハッシュすると空白・改行・ローカル変数リネームで全部無効化する。

- **整形・コメントは無視**してハッシュ (構文木の形のみ)。
- ローカル変数名は **α 正規化** (位置インデックス化) してリネーム耐性を持たせる。
- 公開シンボル名・型は含める (意味が変わるため)。
- 何を無視し何を含めるかが命中率を決める。プロンプトキャッシュの「サイレント無効化犯」問題が
  *再帰的に* 出てくる領域。ここを設計の中心に置く。

**PoC の粒度 = 関数 (確定)**: まず**関数単位**で content-address する。AST 解析で
**関数の追加・更新が一切無ければ同一ハッシュ**とする (整形/コメント/ローカル名のみの変更は
関数 AST が不変 → 同一ハッシュ)。関数本体の構造が変われば別ハッシュ。
これで「整形だけ→不変 / 挙動変化→変化 / 別関数→衝突なし」を関数粒度で先に検証する (§14 PoC)。

### 4.3 アーキテクチャルール (ドメイン契約の一般化)

ドメインを「入出力署名」でなく **スコープ付きアーキテクチャルール** として定義する。

- **ルール = スコープ (global | ドメイン) + 述語 (DAG/KG クエリ) + 重大度 (block | warn)**。
  - **global**: 全実装に当たる (層の依存方向、state アクセス経路、hot path の alloc 禁止 …)。
  - **ドメイン**: そのドメインを実装する全コードに当たる (Skill は Action を emit / Effect は state-write を
    宣言 / Brain は Control を直接呼ばない …)。
- **ある実装が守るべきルール = global ∪ (担うドメインのルール)**。verify はこれをクエリとして走らせ、
  ヒット = 違反。述語の実体は Datalog 風層 (CodeQL/Glean の「コード = 事実 + ルール」)。

**ルールの記述 (2 モード。述語言語は人に書かせない)**:

1. **選択式 (preset カタログ)**: よくあるルールをパラメータ付きで用意 → 選んで埋める。
   例「`{A}` は `{B}` を呼ばない」に A=Effect, B=Control。共通の 8 割はゼロコード。
2. **テンプレートコード (by-example)**: 禁止/推奨の *コード片* を実構文で書く (Semgrep/AST-grep 流)。
   `$SKILL.mutate($STATE)` を禁止パターンに → 構造マッチ。正例・負例の両方。
   既存の良い実装を指して「これが満たすルールを **逆生成**」も可 (リポからブートストラップ)。

- 両モードとも内部の同じ述語エンジンへコンパイル (CodeQL 流は隠す)。
  1 ドメインオントロジー plugin = preset 配置 + template ルール + 手本/カード (§7)。
- 緩さ: 部分適合 + 信頼度を許容。暗黙規約のマイニング → ルール候補提案 → 批准 (§4.5 硬化ループ)。

### 4.4 ドメインカード (Domain Card)

キャッシュ・配信の **単位**。生コードでも生仕様でもなく、LLM が蒸留した正準サマリ。

- 内容: 「何のドメインか / 契約 / 主要コードアンカー / 仕様参照 / 複雑度」。
- **下の code+spec の Merkle ハッシュをキーに 1 回だけ計算** (高い LLM パス) → content-addressed でキャッシュ。
- コード変更 → ハッシュ変化 → **影響したドメインのカードだけ再計算**。
- この 1 個が **AI には高精度文脈束、人間にはドメイン/複雑度サマリ** を兼ねる。

### 4.5 コード ↔ 仕様 3 段リンカ + 硬化ループ

仕様 (spec/*.md, DESIGN.md, feature 仕様) のクレームを、コードアンカーに結びつける。
結合は異種混合 (構造的コード × 自然言語仕様) なので 3 段で当てる。

| 段 | 手法 | 強さ | 鮮度 |
|---|---|---|---|
| 明示 | コードに仕様 ID / 仕様にシンボル参照 (`@implements SPEC-123`, `tier-routing.ts` 等) | 最強・確実 | 安い |
| 構造 | 命名・配置のヒューリスティック | 中 | 中 |
| 意味 | embedding/LLM で段落 ↔ コードを照合 | 弱・曖昧 | 高い |

- 各辺は **信頼度 + 根拠 (明示/構造/意味)** を持つ。「ハードリンクか推測か」が常に分かる。
- embedding は競合する保管庫でなく、仕様ノード内の **リンク信号** として使う。
- **硬化ループ**: 意味リンカが低信頼度の辺を提案 → 人間/CI が批准 → 明示リンク (高信頼度・
  変更時に安く検証可) へ昇格。使うほど決定的になり、キャッシュ命中率が上がる。
  *コード↔仕様の結合が、自己強化するキャッシュになる。*

### 4.6 KG 射影 (traceability クエリ用)

DAG (コンテンツアドレスの不変構造) を *下層の真実* とし、**KG (Kuzu) はその上の問い合わせ用射影**。

- 「§4 を実装する全コード」「System X に触る全仕様」などの traceability クエリは graph 探索で。
- KG は materialized view: DAG から生成され、DAG が真実。可変グラフを別途管理しない。
- 集計・近傍探索は Kuzu、複雑な導出ルール (ドメイン検出の一部) は将来 Datalog 風層を差せるよう
  **query 層を抽象化**しておく (CodeQL/Glean/Soufflé の先行技術を参照)。
- embedding-RAG を核にしない理由: retrieval が揺れ prefix が揺れる → キャッシュと本質的に不適合。

## 5. 動的層: 実行トレース

「今どのフレーム / どのソースにいるか」を監視し、静的 DAG に重ねる。

### 5.1 フレーム同期スコープ計測

ゲームはメインループ = 自然なフレーム境界を持つ。「今どのフレームか」= ループ毎にカウンタ、
「今どのソースか」= アクティブな zone スタック (各 zone が Anchor ID を持つ)。

| | C++ | C# / Unity |
|---|---|---|
| スコープ計測 | Tracy / Optick / Remotery 系 (フレーム整合・source 付き・低オーバヘッド) | ProfilerMarker / CustomSampler (Unity)、`EventSource` |
| 無改変サンプリング | サンプリング + DWARF/PDB で IP→source | EventPipe / dotnet-trace、ICorProfiler enter/leave |

### 5.2 マーカー自動注入

手置きは精密だが面倒、自動計測は楽だがドメインを知らない。**静的解析がドメイン境界を知っている**ため両取り:

- 解析器が「ドメインカードの入口」を知る → **そこにだけマーカーをコード生成/注入**。
- マーカーは **静的 DAG と同じ Anchor ID** を埋め込む → 静的と動的が完全に噛み合う。
- 精密・低オーバヘッド・手置き不要・ドメインタグ付き。

### 5.3 静的ループ骨格 + 動的アニメート

- 静的: `main()` / tick から辿る call graph で「System tick の順序 = あるべきループ構造」を先に描く。
- 動的: 実トレースでそれを *アニメート*。設計 (静的) と実際 (動的) のズレが見える。

### 5.4 同一アンカー縫合

zone(Anchor ID) ↔ ドメインカード を join。
「フレーム N でこの順に System が回った / どのドメインがホットか」が、静的カードに紐付いて見える。

### 5.5 局面の学習 (手法確定 — 実装済 G10)

生トレース (zone 集合) の上に「いまどの *局面* か」を学習する層。静的側 (自動発見 / locate) と
対称に、**オフライン語彙作り (局面発見)** と **オンライン現在地 (局面分類 → §6 You-are-here の phase)**
の 2 問題に割る。静的の硬化ループ (§4.5) と対称な「動的の硬化ループ」。

**手法 (旧「クラスタ vs FSM」「トレース圧縮」未決を確定)**: 二択ではなく段が違うので併用する。

1. **トレース圧縮 = ドメインセット署名 (決定的)**: `stitch.ts` が anchor→domain に畳んだ後、
   フレームを `{アクティブドメイン集合 (+ top-k by time) + hot ドメイン}` に離散化し、
   `signatureId = sha256(正準記述)` を content-addressed なキーにする。ドメイン数は小さく距離計算
   不要で決定的 → 同一トレースは常に同一 phase id (§4.6/§9 のキャッシュ契約を動的側でも守る)。
2. **状態 = 署名のクラスタ (任意 merge のみ)**: 署名が断片化したら Jaccard 類似で *固定しきい値*
   マージするだけ。k-means 等の乱数初期化・K 未知は持ち込まない (決定性を壊す)。
3. **ダイナミクス = FSM induction**: phase id 列のフレーム間遷移を観測 FSM (局面=状態、遷移=辺
   +出現確率、同一 phase 連続=dwell、未到達=デッドコード) として誘導。静的ループ骨格 (§5.3) と
   diff して「あるべき遷移 vs 実際」を出す。
4. **ラベル (LLM 1-shot・cached)**: 各局面を署名キーで 1 回だけ命名 (`card.ts` のキャッシュと対称)。
5. **動的硬化ループ**: 低信頼の境界/merge を提案 → 人/CI 批准 → 明示 phase def 昇格 → DAG/KG 注釈。

実装は `src/dynamic/phase/` (signature/discover/fsm/label/classify) + `viz/where.ts` 配線 (G10)。

## 6. 配信・可視化層

`render_pipeline` plugin の「静的 scan → Web 可視化」前例の *動的版*。

- **Transport**: LUDIARS には既に配信路がある (`ergo_custos` の inAppBridge `/stream` port 17778 +
  Custos の WebRTC/Unity View)。トレースをこの transport に相乗りさせ、ビューアを Web で出す。
  プロセス内リングバッファ → 別プロセス (ビューア) へ stream。低オーバヘッド。
- **ビュー**: フレーム × ドメインタイムライン / 静的 DAG を現在アクティブな Anchor で点灯 (光るグラフ) /
  現在フレーム + アクティブ zone スタックのライブカーソル (You are here)。

## 7. 拡張性 (3 種 plugin)

1. **言語フロントエンド** plugin: tree-sitter grammar 単位 (C++/C#/TS/Rust…)。
2. **ドメインオントロジー** plugin: ゲーム別/ドメイン別のドメイン語彙 = 「契約 + 検出ルール + カード要約テンプレ」。
   - 例: AdventureCube (`Skill→ActiveObject→Action` + beat + Shield/Melee)、
     ergo_character (`Brain/Check/Control/Effect`)、KuzuSurvivors (wave/pickup/upgrade)、
     Memoria 等は汎用 (CRUD/Pipeline/Adapter)。
3. **出力アダプタ** plugin: Web 可視化 / AI 文脈束 / 人間向けレポート。

コア (Parse/Model/Analyze) は不変、上記 3 点で各ゲーム・各言語へ広げる。

## 8. 複雑度メトリクス (game-aware)

cyclomatic だけでは不足。ゲームの複雑度は **システム間結合と状態空間**に出る。

- ドメインオーバーラップ密度 (1 entity に触るドメイン数)
- 共有 state への fan-in
- ドメインをまたぐ依存深さ
- (補助) cyclomatic / fan-in-out / 結合度

メトリクスはすべて DAG/KG 上のグラフ集計として算出 (ドメイン整理と同じグラフの副産物)。
Memoria 肥大化も同じ尺度 (1 モジュールに機能が何個刺さっているか)。

## 9. キャッシュ設計

- **文脈束 = `f(query, DAG スナップショット)` の純関数**。安定ソートで決定的に組む。
- 同じ問い → 同じ束 → prefix 安定 → `cache_read` が効く。
- 変更時は Merkle で影響範囲だけ無効化 (差分索引と地続き)。
- 束の中も「不変 (ドメインサマリ/仕様クレーム/型定義) を前、可変 (今回の問い) を後ろ」に並べる
  → `@ludiars/llm-gateway` の `orderSegments` をそのまま使う。
- **本ツール自体がトークン節約の実装例**になる (前方一致 = 部分木一致の自己相似)。

### 9.1 クリーンコードの supply → verify ループ (重心の中身)

**「クリーン」の定義**: 抽象的な美しさでなく、この codebase の grain (ドメイン・ルール・仕様) に
逆らわないこと。① 既存ドメインを再発明しない ② 結合を無闇に上げない ③ 仕様の意図に結びつく
④ 周りと一貫。九龍城でよい、ただし **増築が既存の柱に噛み合うこと**を保証する (≠ 綺麗に書く)。

**① 生成前 supply (クリーン文脈束)** — タスクに対し DAG/KG が決定的に組む束:

| 要素 | 中身 | 防ぐ不潔 |
|---|---|---|
| 着地点 | §9.1.1 で決定 | 別構造の乱立 |
| 適用ルール | global ∪ ドメインルール (§4.3) + 手本 | 我流の再実装 |
| 仕様束 | 結びつく仕様クレーム (§4.5) | 意図のズレ |
| 規約抽出 | 兄弟コードから抽出した命名/エラー処理/構造 (汎用 style でなく*この場の*流儀) | 浮いたコード |
| 影響半径 | KG 依存クエリ = 壊し得る範囲 | 不注意な破壊 |
| 重複回避 | 「ドメイン M は X に既存、拡張せよ」 | 再発明 (AI 最大の不潔源) |

束は content-addressed (キー = 着地点アンカーの Merkle ハッシュ) → 安定 →
`orderSegments` で不変前・可変後ろ。

**② AI が書く** → 既存ドメイン・ルールに沿った、逸脱しないコード。

**③ 生成後 verify (クリーン判定)** — diff を DAG に再パースし、適用ルール (§4.3) をクエリ実行:

| ゲート | 判定 | 種別 |
|---|---|---|
| ルール適合 | 適用ルール集合に違反なし (§4.3) | block |
| 重複検出 | 既存ドメインカードに似すぎ (ここだけ embedding を*フラグ*に) | block |
| 仕様結合 | 仕様クレームに結びつく。orphan (仕様なし) = 逸脱疑い | warn→block |
| 結合デルタ | 共有 state fan-in / ドメイン跨ぎ依存が *リポ自身の上位 %* 超 (§9.2) | warn |
| 規約逸脱 | 命名/イディオムが兄弟から乖離 | warn |

→ 構造化 verdict (ゲート別 pass/fail + 具体アンカー + 修正示唆)。

**④ ループ** — 不合格 → 違反 (アンカー+ルール+手本) が次の生成 steering に入る。回す度に締まる。
各反復で新規は AI の変更部分木だけ → verify は diff だけ再パース → **ループはほぼキャッシュ読み**。

**MCP 口 (v1)**: `anatomia.context(task)` → 束 / `anatomia.verify(diff)` → verdict /
(補助) `anatomia.where(task)` → 着地点 / `anatomia.impact(anchor)` → 影響半径。

#### 9.1.1 着地点の決定

**着地点 = f(ドメインカテゴリ, 層ルール, 既存兄弟)**。

```
新規 → ドメインで意味論 (これは Effect) → アーキ層ルールで層 (Effect は effect 層)
     → 既存兄弟 (他の Effect) で具体ファイル/書き方をピン
```

- 前例アリ = 決定的。前例ナシ (新ドメイン) = 層まで決まり具体は提案 → 選択が前例化 (硬化)。
- 横断機能 (状態異常 = UI+戦闘+セーブ) = 先にドメイン分解してから各々へ。各着地点は信頼度付き。

### 9.2 閾値 (codebase 相対)

「結合が増えすぎ」を絶対値で決めない (リポの個性を殺す)。**そのリポ自身の分布**から導く。
例: このリポの fan-in 中央値 3 / 上位 5% が 8 → 新コードが上位 5% を超えたら警告。
= 「このリポの普通からの逸脱」で判定。九龍城でも異質なものだけ引っかかる。分布はリポの成長で更新。

## 10. RAG・代替の評価 (なぜこの構成か)

「キャッシュドメイン」前提では、決定的 retrieval が必須。

| 方式 | 精度 | キャッシュ適性 | ドメイン/複雑度抽出 | 評価 |
|---|---|---|---|---|
| Embedding RAG | △ | **✕ (揺れる)** | ✕ | 曖昧な入口の補助のみ |
| 静的解析 DAG/KG | ◎ | **◎ (決定的)** | ◎ | **核** |
| リポマップ | ○ | ○ | △ | 軽量版・補助 |
| LSP 駆動 | ◎ | ○ | △ | 裏取りに使用 |
| エージェント検索 | ◎ | ○ | ✕ | leaf 本文取得に使用 |

- KG は結合の *保管・配信* に最適だが、結合の *生成* は 3 段リンカ (§4.5) 次第。KG 単体を
  「1 番」と見ると本丸 (リンカ) を見落とす。
- → コンテンツアドレス DAG を下層の真実、KG をその射影、embedding を信号、に役割分離。

## 14. 実装方針 (フルセット・No-MVP)

`feedback_full_set_no_mvp` / スキル `full-set-implementation` に従う。
**MVP / 最小縦切りは作らない。設計済み機能をフルセットで実装 → 全部配線 → テストする。**
局面学習 (§5.5) は手法確定 + 実装済 (G10)。残保留は実トレース録画経路の配線のみ。
タスクの全列挙は `Ars/Anatomia/TASKS.md`。

依存順 (全部作る前提の順序): 静的 DAG → グラフ/KG(Kuzu) → ドメイン・ルール → コード↔仕様 →
supply/verify → アダプタ(MCP/CLI/Web) → 動的トレース → 動的可視化。

各 feature で計測: 正規化ハッシュの命中率 / ルールの取りこぼし率 / 束の決定性 / verify の精度。
**実装は Sonnet、最終レビューは Opus。**
