01何をするサービスか

Anatomia は「AI 用 RAG」ではなく、AI のコード生成を綺麗に保つ建築規約オラクル。 ゲームのコードベースを、次の 3 用途に同時に使える知識構造へ変換する。

(a) AI への文脈供給

決定的な文脈束

高精度・キャッシュ安定な「ドメイン単位の文脈束」を決定的に組んで AI に渡す。

(b) 人間の把握

動作構造の可視化

メインループ／システム実行順と複雑度（結合・状態空間）を可視化する。

(c) コード ↔ 仕様

双方向トレーサビリティ

どのコードがどの仕様クレームを実装しているかを双方向に引く。

本質は 静的構造（何のドメインが存在するか）と 動的実行（今フレーム何が火を噴くか）と 仕様 を、 同一の Anchor ID で縫合すること。「クリーン」= 抽象的な美しさではなく、その codebase の grain（ドメイン・ルール・仕様）に逆らわないこと。

02重心：supply → verify ループ

Anatomia は文脈を渡すだけでなく、生成の前後で二度効く。決定的キャッシュのおかげで ループで何度叩いても安い。

① 生成前 (steering)

文脈を供給

「この機能=ドメインM、実装は anchor X,Y、仕様§S、規約Z、綺麗に足す場所W」を AI へ。

② AI が書く

逸脱しないコード

既存ドメインに沿った、grain に逆らわないコードを生成。

③ 生成後 (verify)

5 ゲートで検証

契約・仕様結合・結合度・重複・規約を判定。不合格なら具体的違反を返す。

④ 再生成

締まるまで回す

違反（アンカー+ルール+手本）が次の steering に入る。回す度に締まる。

5 つの verify ゲート

  • ルール適合（block）— global ∪ ドメインルールに違反なし
  • 重複検出（block）— 既存ドメインカードに似すぎ（ここだけ embedding をフラグに）
  • 仕様結合（warn→block）— 仕様クレームに結びつく。orphan = 逸脱疑い
  • 結合デルタ（warn）— 共有 state fan-in / ドメイン跨ぎ依存が リポ自身の上位 % 超
  • 規約逸脱（warn）— 命名/イディオムが兄弟コードから乖離

03設計の柱

3 つの設計原則と、層を貫く 1 本のアンカー ID。

原則 1
キャッシュをデータ構造にする。コンテンツアドレッシング (Merkle) で無効化を構造から無料・正確に得る。
原則 2
静的・動的・仕様を同一アンカー ID で縫う。「存在する／実行された／仕様」が同じ ID で繋がる。
原則 3
結合は類似度でなく適合・証拠で。embedding は補助信号、契約と明示参照を一級市民に。

用語

  • DAG = 正規化 Merkle-AST（関数粒度・acyclic）＝キャッシュ土台。意味が同じなら同一 Anchor ID。
  • KG = その上の派生ビュー（Kuzu 射影、関係クエリ用・循環可）。DAG が真実、KG は再生成。
  • ドメインカード = キャッシュ・配信の単位。LLM が蒸留した正準サマリ（何のドメイン/契約/主要アンカー/仕様参照/複雑度）。

解析パイプライン（analyze()）

parse (tree-sitter)→ Merkle-AST DAG→ graph / KG (Kuzu)→ ドメイン・ルール→ コード↔仕様→ supply / verify→ アダプタ (MCP/CLI/Web)→ 動的トレース

各層の役割と依存関係は アーキテクチャ／グラフビュー でインタラクティブに辿れる。
